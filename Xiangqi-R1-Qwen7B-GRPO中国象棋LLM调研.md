# Xiangqi-R1：Qwen2.5-7B + GRPO 训练中国象棋 LLM 的论文调研

- 论文：[arXiv 2507.12215](https://arxiv.org/abs/2507.12215)（USTC，2025-07）
- 代码：[MINE-USTC/Xiangqi-R1](https://github.com/MINE-USTC/Xiangqi-R1)

## 背景

R1 范式（SFT → GRPO）跑赢数学/代码推理后，自然会问：能不能搬到棋类这种**强空间约束 + 长程策略**的任务上？Xiangqi-R1 是一个比较干净的样本——它把目标定得很克制：**让 LLM 在棋盘上不胡走 + 能判断局势**，没有声称要赢棋。这种"目标降维"本身就是一个值得学的判断。

## 核心问题

通用 LLM 在中国象棋单步上有两个老问题：

1. **走法非法率高**（Doubao-Pro-1.5-Think 的 legal@1 只有 0.77，意味着 1/4 走法直接违规）
2. **没有局势判断**（红方略优 / 黑方大优 五分类只有 0.40）

光做 SFT 解决不了第二个问题——模型会拟合"答案格式"但不真的理解局面强弱。所以才上 GRPO。

## 三阶段训练架构

| 阶段 | 数据规模 | 任务 | 关键配置 |
|---|---|---|---|
| SFT-1 | 5.1M (FEN, 最佳走法) | 学走子合法性 | bs 64, 1 epoch |
| SFT-2 | 52.9K（加入局势 5 分类标签 + 思维链格式） | 学局势判断 + `<Think>/<Answer>` 输出格式 | bs 32, 3 epoch |
| RL-GRPO | 52.8K（每条样本带引擎对**所有合法走法**的估值） | 用 reward 修正错误倾向 | bs 8, 2 epoch, 8 rollouts |

全程 4×A6000 + LoRA (r=16, α=32) + DeepSpeed ZeRO-2，平民配置。

**值得注意**：RL 数据只有 SFT-1 的 1%。GRPO 不靠数据量靠 reward 信号。

## GRPO 奖励函数设计（最值得记住的部分）

总奖励 R = R_format + R_move + R_analysis（max=5）。

### R_move 的分层设计

走法奖励不是"对就 1 错就 0"，而是**三档阶梯**：

```
1. 走法合法 + 中文记法对得上引擎记法 → +1
2. 等于引擎首选走法                    → +1
3. 与引擎最优值差 < 100 centipawn       → +1
```

这是整个工作里最聪明的设计点。要做到第 3 档，前提是**用引擎（Pikafish d25）给所有合法走法打分**，不是只标注一个最优。这样：

- 模型走出"次优但合理"的走法时拿到部分奖励，不会因为没命中 best 就归零
- "good move"（差距 < 100 cp）的判定避免了 reward 过稀疏
- legal / good / best 三档天然形成密集梯度，GRPO 优势函数有足够区分度可学

### R_analysis 的离散化

局势打分（连续 cp 值）按 ±100 / ±800 阈值切成五桶：大优 / 略优 / 均势 / 略劣 / 大劣。把回归转成分类——LLM 友好，奖励稀疏度也好控制。

### R_format

强 regex：必须 `<Think>...</Think><Answer>局势分析:xxxx\n最优走法:xxxx(ICCS)\n</Answer>`，差一个字符就 0。在 R1 范式里几乎是标配。

## 关键结果

vs SFT-Stage2 自家基线（即不上 RL）：

| 指标 | SFT-2 | +GRPO | Δ |
|---|---|---|---|
| legal@1（合法率） | 0.864 | **0.952** | +8.8 |
| 3-class@1（局势粗分类） | 0.431 | **0.629** | +20.0 |
| 5-class@1（局势细分类） | 0.389 | **0.610** | +22.2 |
| best@1（命中引擎首选） | 0.119 | **0.181** | +6.2 |

## 关键洞察

### 1. "降维"目标，避开打不过引擎的死路

论文坦白：**模型仍无法稳定下完整盘棋**。所有指标都是单步评测，没有 vs Pikafish 胜率/Elo。这个老实的边界划得很好——不在自己赢不了的赛道硬碰，转而把"单步合法性 + 局势判断"做扎实，让 LLM-for-Chess 这条路有可量化的进展。

### 2. 让引擎当 reward labeler，而不是当对手

直接做"自我对弈 + 胜负 reward"对 7B 模型几乎不可行（信号过稀疏，整局对弈又崩）。Xiangqi-R1 把引擎用作 **dense reward provider**：每个状态全合法走法估值一次，离线生成 reward labels，把 RL 退化成单步任务。这是 7B 模型现实可行的妥协。

### 3. SFT-2 单独训"局势判断"是被低估的一步

22 个百分点提升里，绝大部分来自 5-class 局势分类，不是走子。**先让模型会"看局面"，再学"会下棋"**——这跟人类学棋顺序一致，但很多 LLM-for-Chess 工作直接跳过这步。

### 4. GRPO 不是万能，靠的是 reward 工程

仓库代码里 `analysis` reward 注册了但**默认没启用**（实际跑 `format + move`），论文表格里却是三种全用。说明 reward 组合需要反复试，作者最后选 R = format + move + analysis 是有 ablation 支持的，不是凭直觉拍。

## 可复用的方法学

抛开象棋这个 domain，这套配方适合任何"有 ground truth 评估器、目标可单步化"的任务：

1. **SFT-1**：海量数据学基础语法/规则合法性
2. **SFT-2**：少量精标数据学高层任务格式（思维链 + 关键标签）
3. **GRPO**：用外部评估器（引擎、编译器、规则系统）给**所有候选**打分，分层 reward 替代二值 reward

关键是第三步——**reward 必须分层 + dense**，否则 GRPO 优势估计崩。
